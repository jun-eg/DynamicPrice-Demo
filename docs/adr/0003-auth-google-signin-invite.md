# ADR-0003: 認証方式 (Google Sign-In × 招待制)

- ステータス: 採用
- 日付: 2026-04-28

## コンテキスト

- 利用者は「価格担当者」など社内の少人数
- 利用者は **共通の Google Workspace ドメインを持たない**(個人 Gmail 等)
- 試験運用なので、認証実装はできる限り薄くしたい(パスワード保管・リセット・MFA 自前実装は避けたい)
- 退職・配属変更時に**即時にアクセスを切れる**必要がある
- 「誰がいつ何をしたか」を監査できる必要がある(`AuditLog`)

## 決定

**Google Sign-In(個人 Google アカウント)× 招待制** を採用する。

### 認証フロー

```
[管理者] 管理画面で「メアド + ロール」を入力 → Invitation レコード作成
   ↓
[利用者] サインインページで「Googleでログイン」クリック
   ↓
Google OAuth で認証(Google が ID・メアドを返す)
   ↓
[アプリ] 取得メアドを Invitation で検索
   ├─ 該当あり・未使用・未失効
   │     → User 作成、Invitation を消化、セッション発行
   ├─ 該当なし or 失効
   │     → ログイン拒否、画面に「招待されていません」表示
   └─ User がすでに存在(2回目以降)
         → User.status を確認、ACTIVEならセッション発行
```

### 技術選択

- **Auth.js (旧 NextAuth)** の Google Provider を使用
- 招待検証は `signIn` callback で実装
- セッションは **JWT**(Cloud Run のステートレス性に合わせる)
- セッション有効期限: 短め(8時間程度)、`updateAge` で延長

### ロール

- `ADMIN`: 招待発行・ユーザー無効化・全機能利用
- `MEMBER`: 価格表示・データ閲覧(招待発行不可)

最初の `ADMIN` は **シードで投入**(自分のメアドを入れる)。以降は招待で増やす。

### 退職・剥奪

- `User.status = DISABLED` にすれば次回サインイン時に拒否
- 既存セッションは JWT 有効期限内は通る(短期間で問題無視)
- 必要なら JWT のセッション検証時に DB 照会で即時無効化(性能との trade-off、必要になったら導入)

## なぜこの選択か

### Google Sign-In(他の認証手段ではなく)

| 候補                        | 評価                                                                                        |
| --------------------------- | ------------------------------------------------------------------------------------------- |
| **Google Sign-In(採用)**  | パスワード管理ゼロ。MFA は Google 側に任せられる。利用者全員が個人 Google アカウントを持つ |
| パスワード(自前)           | パスワード保管・リセット・ハッシュ運用すべて自前。試験運用にコスト過剰                    |
| マジックリンク              | メール送信インフラが必要。Resend 等で可能だが、Google Sign-In の方がさらに薄い              |
| 他のOAuth (GitHub など)     | 全員が持っているとは限らない。Google Gmail はほぼ全員が持っている前提が成り立つ             |
| SAML/OIDC SSO(社内IdP)    | 社内に IdP を持たない                                                                       |

### 招待制(自己申請制ではなく)

- 試験運用で利用者は事前にわかっている(価格担当者・管理者)
- 自己申請にするとスパム対策・申請内容の妥当性判断が必要 → 過剰
- 招待発行 = 管理者が完全コントロール

### Cloud IAP を採用しない

- IAP の典型的な運用は **Google Workspace ドメインで一括許可**
- 個人 Gmail を許可するには **メアドを IAP の Allow List に追加** する運用になり、結局アプリ層の招待管理と二重管理になる
- アプリ層で完結させるほうがシンプル
- 監査ログとロール管理(`ADMIN` / `MEMBER`)もアプリ層で持つほうが自然

### Auth.js を採用する理由

- Next.js (App Router) と公式統合あり
- Google Provider が標準サポート
- セッション形態(DB / JWT)を選べる
- `signIn` callback で招待検証を差し込める

## セキュリティ上の注意

- **招待は短期失効**: `expiresAt` に上限を設ける(7日推奨)
- **CSRF**: Auth.js が標準対応
- **セッション固定**: JWT 化で軽減、定期 rotation を設定
- **メアドの正規化**: Google が返す `email_verified=true` だけ受け入れる
- **ロール詐称**: `User.role` を JWT に焼かず、リクエストごとに DB から引く(セキュリティ vs 性能の判断)

## 監査

以下を `AuditLog` に記録:

- `LOGIN` / `LOGIN_REJECTED`
- `LOGOUT`
- `USER_INVITE`(誰が誰を招待したか)
- `USER_DISABLE`(誰が誰を無効化したか)
- `PRICE_VIEW`(誰がいつ何の日付・部屋タイプを見たか)
- `COEFFICIENT_RECOMPUTE`(誰がいつ係数を再計算したか — 管理操作)

## 影響

- 全利用者は **個人 Google アカウントを持つ前提**
- 招待発行は管理者の手動オペレーション
- セッション・認可ガードは Next.js (web) と NestJS (api) の両方で必要
  - web: Auth.js のミドルウェア
  - api: NestJS の Guard で JWT を検証(web と同じシークレットで署名検証)、ロール認可も Guard / カスタムデコレータで実装
