# ADR-0008: 価格クランプ範囲の保持粒度

- ステータス: 採用
- 日付: 2026-04-30

## コンテキスト

- `02-pricing-model.md` レイヤー1 で「価格上限・下限を必ずクランプ。極端な値段が出ない安全弁」と明記しているが、上下限値の保持先が `03-data-model.md` で未定義のままだった
- `/recommendations` 実装(Step 7)で必須となるため、着手前に保持先を確定する必要があった
- 候補:
  - **案A**: `BasePrice` テーブルにカラム追加(`RoomType × Plan × effectiveFrom` 粒度)
  - **案B**: `RoomType` または `Plan` 単位で保持
  - **案C**: 環境変数 / 設定値で全体一律

## 決定

**案A を採用する**。`BasePrice` テーブルに `priceMin` / `priceMax` カラムを追加する。

### スキーマ

| カラム     | 型              | 制約     | 備考                       |
| ---------- | --------------- | -------- | -------------------------- |
| `priceMin` | `decimal(12,2)` | NOT NULL | 推奨価格の下限(税込)      |
| `priceMax` | `decimal(12,2)` | NOT NULL | 推奨価格の上限(税込)      |

- **CHECK 制約**: `priceMin <= priceMax`
- `amount`(基準価格)との大小関係は制約しない
- ユニーク制約は既存通り `(roomTypeId, planId, effectiveFrom)`

### 運用

- Seed 投入時に基準価格と同時に上下限を決定する
- 初期値の目安: 基準価格 ±30%(運用ヒアリングで上書き)
- 手動編集 UI は MVP では持たない(DB 直接変更)。Phase 2 で UI 化候補

## なぜこの選択か

### 案A を選んだ理由

- **設計思想との整合**: 「極端な値段が出ない安全弁」(`02-pricing-model.md` レイヤー1)は粒度依存で意味が変わる。素泊まりの安価な部屋(例: 8,000 円)と一泊二食付きスイート(例: 80,000 円)で同じ上下限を使うと、安全弁として機能しない。**部屋タイプ × プラン単位**が必須要件
- **既存スキーマと粒度が一致**: `BasePrice` が既に `RoomType × Plan × effectiveFrom` の粒度を持っており、上下限もこの粒度に乗せるのが自然。期間で値を変えたい運用も `effectiveFrom/To` で表現できる
- **migration コストが実質ゼロ**: Step 1(初回 migration)着手前に決定したため、追加 migration ではなく `init` migration に含められる
- **運用上の凝集度**: 基準価格と上下限は「基準 ±X%」のようにセットで決まる性質のもの。同じテーブルにある方が seed・レビュー・管理が楽

### NOT NULL を採用する理由

- NULL 許容にすると「未設定 = 無限大」の解釈分岐が必要になり、推奨価格計算ロジックが複雑化
- MVP では明示的に値を入れる方が、安全弁としての意図が明確

### `priceMin <= priceMax` のみ CHECK 制約に入れる理由

- 自明な不整合(下限 > 上限)を DB レイヤで弾く
- `amount` との大小関係まで縛ると、運用上「基準価格を一時的に上下限の外に置きたい」(例: 試験的なキャンペーン価格を基準にしてクランプは保守的に効かせる)ケースで動けなくなる

## 検討して見送ったもの

### 案B: `RoomType` または `Plan` 単位

- 同じ部屋タイプでも素泊まりと一泊二食では価格レンジが大きく異なる(食事原価差) → `RoomType` 単位では粒度不足
- 同じプランでも部屋ランクで価格レンジが異なる → `Plan` 単位でも粒度不足
- 結局 `RoomType × Plan` の粒度が必要になり、案A に劣る

### 案C: 環境変数 / 設定値で全体一律

- 安価な部屋と高価な部屋に同じ上下限を適用すると安全弁として機能しない
- 試験運用フェーズで一旦動かしても、運用に入った直後に「個別に持ちたい」となり、案A への移行コストが発生する
- MVP の最速性は得られるが、設計思想との不整合コストの方が大きい

## 影響

- `03-data-model.md` の `BasePrice` テーブル定義を更新済み
- Step 1(Prisma スキーマ)の初回 migration に `priceMin` / `priceMax` を含める
- Step 2(seed)で `BasePrice` 投入時に初期値を決定する
- Step 7(`/recommendations`)で `clampPrice(rawPrice, priceMin, priceMax)` を呼び出す
