# ADR-0007: 係数推定式の確定(単価ベース統一)

- ステータス: 採用
- 日付: 2026-04-30

## コンテキスト

- `02-pricing-model.md` で「単純平均または線形回帰」と曖昧な記述のままだった
- MVP では透明性最優先、説明可能性が損なわれない式を選ぶ必要がある
- 候補は大きく2系統:
  - **件数ベース**: その月/曜日/ビンの予約件数 ÷ 平均件数
  - **単価ベース**: その月/曜日/ビンの平均1泊単価 ÷ 全期間の平均1泊単価

## 決定

**全係数を「成立予約の1泊単価平均」ベースで統一**する。

### 1泊単価の定義

```
1泊単価 = totalAmount / nights
```

- 対象: `cancelDate IS NULL` の Reservation のみ
- 人数による按分はしない(部屋単位の値を見たい)
- `totalAmount` は元CSV の値そのまま(税込前提、変更しない)

### 各係数の式

```
季節係数(月M)            = avg(1泊単価 | checkInDate.month = M)     / avg(1泊単価, 全期間)
曜日係数(曜日D)          = avg(1泊単価 | checkInDate.dow = D)       / avg(1泊単価, 全期間)
リードタイム係数(ビンB)  = avg(1泊単価 | leadTimeBin = B)            / avg(1泊単価, 全期間)
```

- リードタイム = `checkInDate - bookedDate`(日数)
- ビン: `0-3 / 4-7 / 8-14 / 15-30 / 31+`(`PriceCoefficient.key` の既存定義に合わせる)
- 連泊予約は **1 レコード = 1 行**で集計(夜単位に展開しない)。チェックイン日に紐付ける
- 推定対象期間: 直近 **24ヶ月**(古すぎるデータは需要構造が変わっている可能性)

### サンプル不足の扱い

- あるキー(月・曜日・ビン)に対して**サンプル数が 30 件未満**なら係数 = `1.0`(効かない)
- 画面では「データ不足」と注記表示

### 推定の保存

- 1 回の推定実行で `(SEASON × 12) + (DAY_OF_WEEK × 7) + (LEAD_TIME × 5) = 24 行` の `PriceCoefficient` を作成
- 全行で同じ `computedAt` を持つ
- 自動推定時の `source` は `unit_price_avg_v1`
- ADMIN が画面から手動保存した場合は `source = 'manual'` で新バッチを作成する。推奨価格計算は最新 `computedAt` を参照するため保存後即反映される

### 外れ値処理

- MVP では実施しない(透明性最優先)
- 必要が出たら ADR を追記する(パーセンタイルカット等)

## なぜこの選択か

### 単価ベース統一(件数ベースではなく)

- **旅館は部屋数が小さい**ため、件数ベースは「物理的に枠が埋まりやすかっただけ」を需要強度として誤計上しやすい
- 単価ベースなら「その時期に高い値段でも売れていた」という需要強度の代理になる
- 季節・曜日・リードタイムを同じ式で扱えて、画面・ロジックの一貫性が高い
- リードタイム係数は「直前期は高単価で売れる」という旅館の実態を素直に表現できる

### サンプル不足は 1.0 フォールバック

- 少数サンプルで暴れた係数を出すと、推奨価格が極端になり安全弁(クランプ)に依存することになる
- 1.0 にしておけば「効かない」状態で安全側に倒せる
- 30 件は試験運用フェーズの目安。データが増えたら見直す

### 外れ値除去をしない

- MVP は「画面に出すだけ」で価格反映は人手
- 担当者が異常値に気付いたら判断で除外できるため、システム側で複雑な前処理を入れる動機が弱い
- 外れ値処理を入れると「なぜこの値が落ちたか」を説明する仕組みが追加で必要になる

### 連泊を夜単位に展開しない

- 展開するとレコード数が膨らみ、`Reservation` 1行 = 1レコードという素朴な対応関係が崩れる
- 稼働率・ADR の式とも揃う(`02-pricing-model.md`)
- 旅館の予約は 1〜2泊が大多数で、按分しなくても歪みは小さい

## 検討して見送ったもの

### 件数ベース(予約件数 / 平均件数)

- 旅館は部屋数が小さく、容量制約と需要強度を分離できない
- 平日と休日で部屋数が変わるわけではないのに、件数比は容量に影響される

### 線形回帰(複数変数同時推定)

- 説明変数間の交絡(季節と曜日と繁忙期フラグ)を扱うには規模が大きい
- 試験運用のスコープ外。Phase 2 候補

### 1泊単価を人数で按分

- 子供料金・幼児料金の按分が原データに依存し恣意的になる
- 「部屋単位の単価」が価格運用上の自然単位

## 影響

- 係数再計算ジョブ(`POST /admin/coefficients/recompute` または `npm run coef:recompute`)で上記式を実装
- `PriceCoefficient.source` は自動推定時 `unit_price_avg_v1`、手動保存時 `manual`
- ADMIN は係数画面から推定値をベースに任意の値へ上書きできる(`PUT /admin/coefficients`)
- 画面では `value = 1.0` のキーは「データ不足」と注記表示
