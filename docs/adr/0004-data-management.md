# ADR-0004: データ管理方針

- ステータス: 採用
- 日付: 2026-04-28

## コンテキスト

- 元データ: 既存PMSからエクスポートした予約履歴 CSV(`ReservationTotalList.CSV`、Shift-JIS、約10万件想定)
- 元CSVには **個人情報(氏名・電話・住所・メールアドレス・自由記述メモ等)が含まれる**
- 利用者は個人デバイスで開発・運用する場面がある
- 試験運用なので、データ取り込みは **開発者が初回1回だけ手動実行** で完了する想定
- リアルタイムなPMS連携は無し(`01-overview.md`)

## 決定

### 1. 元CSV はリポジトリに含めない

- `data/raw/` を `.gitignore` 対象とする
- `*.csv` `*.parquet` `*.duckdb` `*.sqlite` を全リポジトリで除外
- `data/raw/.gitkeep`、`data/processed/.gitkeep`、`data/README.md`、`data/sample.csv` のみ git 管理(ホワイトリスト)

理由:
- リポジトリ肥大化を避ける
- 個人情報の事故的なコミットを防ぐ
- Single Source of Truth の原則は「設計・決定・手順」が対象であり、データ実体は別管理(認証情報と同じ「例外」枠)

### 2. 元CSV の保管は開発者ローカルのみ

- 配置先: `<project>/data/raw/ReservationTotalList.CSV`
- クラウドストレージ(S3等)には保管しない(`02-deployment-cloud-run.md`)
- 開発者間でのデータ受け渡しは別途、安全な手段で(個人情報を含むため)

### 3. PII(個人情報)は **DB に投入しない**

シードスクリプトで以下を **完全に落とす**(NULL でも入れない、列ごと除外):

```
宿泊者氏名 / 宿泊者氏名カタカナ
予約者氏名 / 予約者氏名カタカナ
電話番号 / 郵便番号 / 住所1 / メールアドレス
会員番号 / 法人情報
備考1 / 備考2 / メモ
```

備考・メモは自由記述で個人情報が混入する典型箇所。価格モデルにも不要なため落とす。

### 4. データ取り込みは初回シードのみ

- `scripts/seed/import-reservations.ts` を1回だけ実行
- アプリケーション側に **CSV取り込みUI を作らない**
- 元CSVの再取得・追加データの反映が必要になった時点で別 ADR で検討

### 5. シード処理の中身

`scripts/seed/import-reservations.ts` がやること:

1. `data/raw/ReservationTotalList.CSV` を Shift-JIS から UTF-8 に変換して読み込む
2. PII カラムを除外
3. `部屋タイプ名称` `商品プランコード` から `RoomType` `Plan` のマスターを生成(正規化)
4. 残りを `Reservation` テーブルに投入
5. `BasePrice` の初期値も別シードで投入(担当者ヒアリングで設定)
6. 最初の `ADMIN` ユーザーを別シードで投入(自分のメアド)

シードは **冪等(同じデータで再実行しても破壊しない)** にする(MERGE/UPSERT)。

### 6. サンプルデータ(コミット可能)

- `data/sample.csv` に **匿名化した数十行** を保持
- テスト・CI・新規開発者のオンボーディング用
- 生成スクリプト: `scripts/seed/make-sample.ts`(元CSVから個人情報をマスキングして抽出)

### 7. 加工済みデータ(`data/processed/`)

- 当面は **使わない**(直接 DB に入れる方針のため)
- 将来、ETL の中間生成物が必要になったら使う想定で空ディレクトリを保持

## 開発フロー

```
[開発者]
  1. data/raw/ReservationTotalList.CSV を配置(自分だけ持つ)
  2. docker compose up -d (ローカル Postgres 起動)
  3. npm run db:migrate
  4. npm run db:seed-master      # RoomType / Plan / BasePrice / ADMIN
  5. npm run db:seed-reservations # 元CSVを読んで PII 除去して投入
  6. npm run dev (Next.js / NestJS 起動)

[本番環境(Cloud SQL)]
  1. ローカルから Cloud SQL Auth Proxy 経由で接続
  2. 同じシードを 1 回だけ実行
  3. 以降は触らない(試験運用中)
```

## なぜこの選択か

### CSV取り込みUIを作らない

- 試験運用では取り込みは1回だけ
- UI 実装(アップロード・進捗・エラー処理・冪等性)はコストに見合わない
- 開発者の手元で完結させる方が安全(本番環境にCSVをアップロードしない)

### PII を DB に入れない(マスキングではなく除外)

- マスキング(ハッシュ化等)は再構成リスクと運用コストがある
- 価格モデルに使わないカラムを入れる動機がゼロ
- 「最小権限」の徹底 = 持たないのが最強

### クラウド保管(S3等)しない

- 開発者・利用者ともに少人数、共有データ基盤の必要性が薄い
- 個人情報を含むデータをクラウドに置く運用責任を負いたくない(試験運用フェーズで)

### 元CSV を git に入れない

- サイズ・履歴肥大化
- 個人情報の事故防止
- リポジトリと「データ」を明確に分離するため(設計思想と一貫)

## 影響

- データを持っている開発者(マスター)が最低 1 人必要
- マスターのマシンが故障したら元データ消失リスク → 個人で別途バックアップを推奨
- 新メンバーが増える場合は、データの受け渡し手順を `runbooks/` に記載

## 検討して見送ったもの

### 開発者全員が data/raw/ に元CSVを持つ

- 個人情報の流通範囲が広がるためNG

### S3 / GCS にマスターを置く

- IAM 設定 + バックアップ + 監査ログ運用が必要
- 試験運用フェーズでは過剰
- 必要になったら別 ADR で復活検討

### Git LFS

- リポジトリにデータが入る前提になり、PII リスクは消えない
- 個人開発・試験運用に不要
