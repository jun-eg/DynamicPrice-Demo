# 01. 全体像とスコープ

## このツールが何か

旅館の **価格担当者向け** 内製ツール。過去の予約データから動的価格(推奨価格)を計算し、画面で提示する。
価格反映そのものはツール外で人が行う(既存PMS・予約サイトへの入力)。

## 試験運用の前提

本リポジトリの設計は **社内試験運用** を目的としている。本番運用と区別する重要な含意:

- 高可用性・SLA要件はない
- 大量同時アクセスは想定しない(価格担当者・少人数)
- データ更新頻度は低い(初回シード後は手動運用)
- 機能は最小、後から拡張する前提で設計する

## スコープ

### スコープ内(MVP)

- **推奨価格の表示**: 対象日 × 部屋タイプ × プランのマトリックスで画面表示(内訳が見える)
- **過去データの可視化**: 月別稼働率・ADR・リードタイム分布など、価格判断の参考情報
- **係数の自動推定**: 過去予約から季節・曜日・リードタイム係数を自動算出
- **係数の手動編集**: ADMIN が推定値をベースに任意の値へ上書き保存できる(`source = 'manual'`)
- **ユーザー管理**: 招待制 / Google Sign-In / 監査ログ

### スコープ外

- 顧客向け予約画面(本ツールは対外公開しない)
- 既存PMS・予約サイトとのAPI連携(値は人が転記)
- CSV取り込みUI / CSVエクスポート(初回シードは開発者が直接投入、出口は画面表示のみ)
- 価格自動反映・スケジュール実行
- 基準価格の手動編集UI(Phase 2)
- 承認ワークフロー(試験運用にとって過剰)
- ML(機械学習)モデル(Phase 2 以降)

## 利用者像

| 役割     | 何をするか                                       | 権限                       |
| -------- | ------------------------------------------------ | -------------------------- |
| `admin`  | 招待発行・ユーザー管理・全機能利用               | 全機能                     |
| `member` | 価格係数の設定・シミュレーション・データ閲覧     | 招待発行とユーザー管理を除く |

(現時点で `member` がメイン利用者。`admin` は管理目的の少数。)

## 既存システムとの関係

本ツールは **既存PMS / 予約サイトの脇に立つ価格決定ツール**(値付けエンジン)。
予約・在庫の管理は既存PMSが担い、本ツールはそこからエクスポートされた予約履歴を読み込んで価格を計算する。

```
[既存PMS]──CSV手動エクスポート──▶[本ツール:DBに初回投入]──▶[推奨価格を画面表示]
                                                                    │
                                                                    ▼
                                                      [価格担当者が見て既存PMSに転記]
```

## システム構成(本番:Cloud Run on GCP)

```
[ブラウザ]
   │ HTTPS
   ▼
[Cloud Run: web (Next.js)]      ── Auth.js + Google Sign-In
   │ 内部呼び出し
   ▼
[Cloud Run: api (NestJS)]       ── 価格計算ロジック・データAPI
   │ Cloud SQL Auth Proxy
   ▼
[Cloud SQL: PostgreSQL]         (asia-northeast1)
[Cloud Logging]                 監査ログ・操作ログ
```

**意図的に外したもの**:

- **Cloud Storage**: ファイル取り込み・エクスポート機能がないため不要
- **Cloud Scheduler / Cloud Run Jobs**: バッチ処理がないため不要
- **Cloud IAP**: 利用者が共通のGoogle Workspaceドメインを持たないため、アプリ層で認証

これらは将来必要になったときに追加する。

## 技術スタック

詳細根拠は ADR を参照。

| レイヤー   | 採用                                  | 詳細                            |
| ---------- | ------------------------------------- | ------------------------------- |
| 言語       | TypeScript (strict)                   | フロント・バック共通             |
| フロント   | Next.js (App Router)                  | SSR + 管理画面                   |
| バックエンド | NestJS                                | REST API(モジュール構造で価格計算・認証ガード等を分離) |
| ORM        | Prisma                                | スキーマ管理・migration         |
| DB         | PostgreSQL                            | Cloud SQL(本番)/ Docker(開発) |
| 認証       | Auth.js (Google Provider) + 招待制    | アプリ層で実装                   |
| モノレポ   | npm workspaces                        | apps/ / packages/ 分離           |
| Lint/Format | ESLint + Prettier                     | ルートに共通設定、CI で必須チェック |
| CI/CD      | GitHub Actions                        | PRで検証、main merge で Cloud Run へデプロイ |
| デプロイ   | Cloud Run                             | web/api を別サービスとして配置   |

## 依存している前提

これらが崩れると設計を見直す必要がある:

- 利用者は社内の少人数(同時アクセス数〜十数)
- 個人Googleアカウントを持つ(Sign-In可能)
- 価格反映はツール外で人手転記(API連携不要)
- 過去予約データはCSVで取り出せる(既存PMSが対応している)
- 試験運用範囲(本番SLAは負わない)
