# 02. 価格決定モデル

## 設計思想

価格は **分解可能な式** で表現する。「なぜこの値か」を説明できることを最優先にする。
ブラックボックスのML一発で価格を出すのは、業務ツールとして説明責任が果たせない・調整不能になるため避ける。

価格 = `基準価格 × 補正係数の積` の形を貫く。

## 全体式

将来の本番運用で目指す姿:

```
推奨価格 = 基準価格(部屋タイプ × プラン)
           × 季節係数(月 / 繁忙期フラグ)
           × 曜日係数(平日 / 金土 / 日祝)
           × リードタイム係数(申込からチェックインまでの日数)
           × 在庫消化補正(予約ペース vs 想定ペース)
           ※ 価格上限・下限でクランプ
```

**MVP の式**(現在の在庫状況を持たないため、在庫消化補正は外す):

```
推奨価格(MVP) = 基準価格 × 季節係数 × 曜日係数 × リードタイム係数
              ※ 価格上限・下限でクランプ
```

各係数は独立に検証できる。係数が `1.0` なら効かない、`> 1.0` で値上げ、`< 1.0` で値下げ。

## 3層ハイブリッド

### レイヤー1: ルール(安全弁)

人間が固定で決める領域。

- **基準価格**: 部屋タイプ × プランごとに手動設定。市場感・原価・経営判断で決める。
- **価格上限・下限**: 算出結果を必ずクランプ。極端な値段が出ない安全弁。
- **繁忙期固定**: GW・お盆・年末年始は「動的にしない」運用も可能。フラグで切替。

### レイヤー2: データ駆動係数(過去予約から自動推定)

過去予約データから「需要の偏り」を推定して係数化する。
**MVPでは単純平均または線形回帰で推定し、結果をそのまま使う**(透明性最優先)。

| 係数             | 推定方法                                             |
| ---------------- | ---------------------------------------------------- |
| 季節係数         | 月別の予約密度 ÷ 年平均 で正規化                     |
| 曜日係数         | 曜日別の予約密度 ÷ 全曜日平均                        |
| リードタイム係数 | 「申込日 → チェックイン日」までの日数別の予約曲線    |

推定結果は `PriceRule` テーブルに保存され、画面では **算出された係数と推奨価格をそのまま表示する**。
レビュー・承認ワークフローは持たない(試験運用にとっては過剰)。

将来、画面から係数を手動上書きする機能を入れる余地はあるが Phase 2。

### レイヤー3: 在庫消化補正(MVPではPhase 2 扱い)

予約状況を見て補正する係数。本格運用で必要な機能。

```
在庫消化補正 = f(残室率, リードタイム)
```

- 残室率が想定より低い(売れている) → 値上げ
- 残室率が想定より高い(売れていない) → 値下げ
- リードタイムが短いほど反応を強くする(直前期は感応度が高い)

**MVPには実装しない**。理由:

- MVPで持つデータは過去予約履歴のみで、現在(将来日付に対する)在庫状況がない
- 既存PMSとの連携で「現在の在庫」を取得できるようになって初めて成立する
- 試験運用の主目的は「過去データから素朴な推奨価格を出すこと」で、レイヤー3は後付けで追加可能

## 含めない要素(意識的に)

| 要素           | 含めない理由                                                          |
| -------------- | --------------------------------------------------------------------- |
| 価格弾力性     | 観察データだけでは交絡が強く、因果推定が困難。Phase 2以降の課題。     |
| 競合価格       | スクレイピング運用が重い・法的リスク。試験運用では不要。              |
| 天候・イベント | データ取得・統合の手間に対し、効果検証が試験段階では難しい。          |
| ML残差補正     | 透明性が落ちる。レイヤー1〜3で説明力が足りないと判明したら検討。      |

## 旅館特有の制約の扱い

| 制約                      | 扱い                                                                |
| ------------------------- | ------------------------------------------------------------------- |
| 部屋タイプ別価格          | 基準価格を部屋タイプごとに保持。係数は共通。                        |
| 平日/休日格差             | 曜日係数で表現(旅館はここが極端)。                                |
| 季節性                    | 月別係数 + 繁忙期フラグ(GW・お盆・年末年始)。                     |
| リードタイム              | 直前割 / 早期割を係数で表現。                                       |
| 一泊二食付き / 素泊まり   | プラン別に基準価格を持つ(食事原価が異なるため係数では表現しない)。 |
| 連泊割                    | Phase 2。MVPでは1泊単価ベースで計算。                              |
| 定員別                    | Phase 2。MVPでは大人2名想定の固定価格。                            |
| リピーター割・団体割      | スコープ外。動的価格と直交する別レイヤー。                         |

## 出力(MVPで画面に出すもの)

主目的は **「基本的な価格指標を担当者が見られる」** こと。それ以上のことはやらない。

| 画面                       | 内容                                                                       |
| -------------------------- | -------------------------------------------------------------------------- |
| 推奨価格マトリックス       | 対象日 × 部屋タイプ × プラン の推奨価格を表で表示。セルから内訳が見える。 |
| 過去データ可視化           | 月別稼働率・ADR・リードタイム分布など、価格判断の参考情報                  |
| 係数一覧(参照のみ)        | 自動推定された季節・曜日・リードタイム係数のグラフ表示                     |

担当者はこれを見て、判断のうえ既存PMS・予約サイトに人手で反映する(ツールは何もしない)。

## 将来の改善余地

ADR への追記対象として、以下は試験運用後に検討する。

- 係数の手動上書きUI(Phase 2)
- 在庫消化補正の連続関数化・UIチューニング(Phase 2)
- 推奨価格と実績乖離のレポート機能
- 価格弾力性の推定(ML/因果推論を含む。データと運用が成熟してから)
